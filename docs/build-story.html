CHAPTER 01项目诞生

2026 年 5 月 21 日清晨，一个想法从脑海中浮现：做一个帮助用户自律打卡、追踪坚持天数、分享心得的 Web 应用。没有团队，没有设计稿，只有一个目标——尽可能快地从零做出一个可运行的全栈产品。

从原型开始

第一步不是写代码，而是用一个独立的 JSX 文件 jieluxia-ui.jsx（567 行）快速搭建 UI 原型。这个文件包含两个组件：WebDashboard（桌面端）和 MobileApp（手机框架），全部使用硬编码数据。目的是验证视觉风格——深海蓝玻璃拟态（Glassmorphism）。

技术选型决策

为什么选 Next.js 15？

App Router 的文件系统路由 + API Routes 可以前后端一体，不需要单独部署后端。一个人开发，全栈框架是最高效的选择。

为什么选 Cloudflare？

D1 数据库（SQLite at edge）免费额度大，Pages 部署简单，全球边缘网络让中国用户也能快速访问。配合 OpenNext 适配器可以把 Next.js 部署到 Cloudflare Workers。

为什么不用 Vercel？

Vercel 的免费额度相对紧张，且国内访问速度不稳定。Cloudflare 在中国大陆有节点，延迟更低。

为什么自研 JWT？

Cloudflare Workers 环境对 Node.js 原生模块有限制。Web Crypto API 是 Workers 原生支持的，用它实现 PBKDF2 + HMAC-SHA256 可以零依赖完成认证，避免了 jose 库的兼容性问题。

CHAPTER 05认证系统

完全基于 Web Crypto API 自研，零第三方依赖。适配 Cloudflare Workers 的 V8 隔离环境。

密码哈希 — PBKDF2

export async function hashPassword(password: string): Promise<string> {
  const salt = crypto.getRandomValues(new Uint8Array(16));
  const key = await crypto.subtle.importKey("raw", enc.encode(password), "PBKDF2", false, ["deriveBits"]);
  const bits = await crypto.subtle.deriveBits(
    { name: "PBKDF2", salt, iterations: 100000, hash: "SHA-256" },
    key, 256
  );
  return `${toHex(salt)}:${toHex(new Uint8Array(bits))}`;
}
  • 算法: PBKDF2 with SHA-256
  • 迭代次数: 100,000
  • 盐: 16 字节随机值，存储为 hex
  • 输出: 256 位（32 字节），存储为 salt_hex:hash_hex

JWT 签名 — HMAC-SHA256

export async function signToken(userId: string, secret: string): Promise<string> {
  const h = b64url(JSON.stringify({ alg: "HS256", typ: "JWT" }));
  const p = b64url(JSON.stringify({
    sub: userId,
    iat: (Date.now() / 1000) | 0,
    exp: Date.now() / 1000 + 2592000,  // 30 天
  }));
  const key = await crypto.subtle.importKey("raw", enc.encode(secret), { name: "HMAC", hash: "SHA-256" }, false, ["sign"]);
  const sig = await crypto.subtle.sign("HMAC", key, enc.encode(`${h}.${p}`));
  return `${h}.${p}.${b64url(new Uint8Array(sig))}`;
}

认证流程

注册 POST /api/auth/register ├─ 接收 email + password + nickname ├─ hashPassword(password) → salt:hash ├─ INSERT INTO users (id, email, password_hash, nickname) └─ signToken(userId) → Set-Cookie: token=xxx; HttpOnly; Secure; SameSite=Lax 登录 POST /api/auth/login ├─ SELECT * FROM users WHERE email = ? ├─ verifyPassword(password, stored_hash) └─ signToken(userId) → Set-Cookie 鉴权 GET /api/auth/me ├─ 从 Cookie 提取 token ├─ verifyToken(token, secret) → userId └─ SELECT * FROM users WHERE id = ? 路由守卫 middleware.ts ├─ 跳过 /api/* 路由 ├─ /login, /register → 已登录则重定向 /dashboard └─ 其他路由 → 未登录则重定向 /login

copy-assets.mjs

将 OpenNext 构建产物中的依赖目录复制到 assets 目录，解决 Cloudflare CI 找不到 _worker.js 依赖的问题。

// scripts/copy-assets.mjs
import { cpSync, copyFileSync, mkdirSync, existsSync } from "fs";

const src = ".open-next";
const dest = ".open-next/assets";

mkdirSync(dest, { recursive: true });
copyFileSync(`${src}/worker.js`, `${dest}/_worker.js`);

for (const dir of ["cloudflare", "middleware", ".build", "server-functions"]) {
  const from = `${src}/${dir}`;
  if (existsSync(from)) {
    cpSync(from, `${dest}/${dir}`, { recursive: true });
  }
}

gen-routes.mjs

生成 _routes.json，告诉 Cloudflare Pages 哪些路径应该走 Worker（服务端渲染），哪些直接走 CDN（静态资源）。

// scripts/gen-routes.mjs
const routes = {
  version: 1,
  include: ["/*"],                              // 默认所有路由走 Worker
  exclude: [
    "/_next/static/*",                          // 静态资源走 CDN
    "/BUILD_ID",
    ...topLevelFiles.map(f => `/${f}`),          // favicon, SVGs 等
  ],
